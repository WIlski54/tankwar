# Tank Wars v0.1 — Design

- **Date:** 2026-06-28
- **Status:** Approved (ready for implementation plan)
- **Author:** Stephan + Claude
- **Source asset:** `Meshy_AI_Crimson_Circuit_Tank_0628104638_texture2.glb` (Meshy AI "Crimson Circuit Tank")

## Goal

Build a playable third-person Tank Wars starter in Three.js. The centerpiece is a
correctly animated set of **running tracks**: the road-wheels spin and the track
**belt visibly runs** on both sides, synced to driving speed, with differential
behaviour in turns. On top of that, v0.1 is drivable and can shoot.

## Findings from the source GLB

Probed directly (`scratchpad/probe.py`, gltf-transform inspect):

- **One single merged mesh**, no separate parts, no animations. 580,630 vertices /
  1,052,874 triangles. Generator: pygltflib. ~40 MB.
- **One PBR material** on a shared 2048² atlas (baseColor, normal,
  metallic-roughness, emissive).
- **Axes:** bbox X ±0.95 = **length** (front↔back), Z ±0.49 = **width** (the two
  track sides sit at +Z and −Z), Y ±0.42 = **up**. Wheel axles run along **Z**.
- **Running gear is cleanly localized:** lower/outer band (Y < ~0.04, |Z| > 0.30)
  holds ~252k verts, split almost perfectly **left (Z>0) 126k / right (Z<0) 125k**.
  So separating the two sides — and wheels from belt — is feasible geometrically.
- **The baked texture cannot be scrolled.** The atlas is a scattered layout: the
  track-band UVs are sprinkled across the entire 0–1 space (PCA elongation ~1.2;
  density map shows dots everywhere, not a strip). Scrolling these UVs would smear
  random atlas regions across the tracks. **Therefore the belt gets fresh
  arc-length UVs + a generated tread.**

### Assumptions to verify during implementation
- **Front direction** (which end the barrel points). Confirm with an offline render
  before finalizing turret/aim math.
- **Turret/barrel segmentation** from the merged mesh (upper-central mass + thin
  forward protrusion). If clean separation proves messy, fall back to a fixed
  forward-firing cannon with camera-relative aim (see Risks).

## Architecture

Two parts: an offline **build step** (Python) that produces a game-ready GLB, and
the **runtime** (Three.js) that loads and animates it. This mirrors the proven
propeller workflow and keeps geometry processing out of the browser.

### Part 1 — Build step: `build_tank.py`
Run once; deterministic; re-runnable. Responsibilities:

1. Parse the merged GLB (POSITION / NORMAL / TEXCOORD_0 / indices).
2. **Segment** triangles into parts by geometry:
   - `Hull` — everything not in another part (static body).
   - `Turret` — upper-central mass above the hull; `Barrel` — thin forward
     protrusion at turret height. (Verify via render; may be merged with Turret in
     v0.1 if pitch is out.)
   - `WheelsLeft` / `WheelsRight` — circular wheel clusters per side, with computed
     **axle center** (Y0, plus the side's Z) so each wheel group rotates cleanly
     around an axis parallel to Z.
   - `BeltLeft` / `BeltRight` — the track-band surface minus the wheels, per side.
3. For each belt: discard atlas UVs and **generate arc-length UVs** — U = position
   around the track loop (parameterized by angle/arc-length about the side's loop
   center in the X–Y plane), V = across belt width. U tiles (repeats N times),
   V ∈ [0,1].
4. Write `assets/tank.glb` with **named nodes** (`Hull`, `Turret`, `Barrel`,
   `WheelsLeft`, `WheelsRight`, `BeltLeft`, `BeltRight`). Belt primitives carry a
   distinct material named **`TrackBelt`** so the runtime can find/replace it.
   Wheel/turret nodes are translated so their pivot is the rotation axis, with a
   compensating node translation (same trick as the propeller hub).
5. Keep original PBR textures on hull/turret/wheels.

**Interface (output contract):** a single GLB whose scene graph exposes the seven
named nodes above; belts use material `TrackBelt`; all parts in original world
positions when transforms are identity.

### Part 2 — Runtime: `index.html` (Three.js via CDN)
Single self-contained page + `start_server.bat` (local server; GLB needs CORS, no
double-click). Modules (kept as small, focused functions/objects):

- **Loader/binder** — load `tank.glb`, resolve the seven named nodes into a `parts`
  object; assert each exists (fail loudly with a console message if a name is
  missing).
- **Belt tread material** — build a **CanvasTexture** (repeating cleats/links drawn
  on a `<canvas>`), `wrapS = wrapT = RepeatWrapping`, applied to the `TrackBelt`
  material. Tread scroll = `offset.x`.
- **Drive model** — state: `speed`, `heading`. Inputs set throttle/steer; integrate
  position over the ground. Compute per-side track linear speed
  (`vLeft`, `vRight`) from `speed` ± steering (differential).
- **Track animator** — each frame: `wheels{Left,Right}.rotation` += `v{side}*dt*kW`;
  belt `offset.x` += `v{side}*dt*kB`. Left/right independent → differential visible.
- **Turret/aim** — mouse → aim point on ground plane (raycast) → `Turret.rotation.y`
  lerps toward aim yaw. (Barrel pitch optional, default out for v0.1.)
- **Weapon** — left-click fires from barrel tip if off cooldown: spawn shell
  (fast-moving mesh/tracer), muzzle flash sprite, WebAudio boom; shell travels,
  tests against targets, spawns impact effect.
- **Targets** — a handful of static destructible objects (e.g. crates/blocks);
  on hit → destroy effect + removal.
- **Camera** — third-person chase behind the hull, slightly elevated, easing toward
  position/look; biased toward aim direction.
- **Environment** — flat textured ground plane, simple sky/fog, blob shadow under
  the tank, basic HUD (crosshair, reload indicator).
- **Game loop** — `requestAnimationFrame`: read input → drive → animate tracks →
  update turret/weapon/targets → camera → render.

## Data flow

```
input (keys/mouse) ─▶ drive model (speed, heading, vLeft, vRight)
                         │
                         ├─▶ tank world transform (move/turn over ground)
                         ├─▶ track animator ─▶ wheels rotation + belt offset
                         └─▶ camera follow
mouse ray ─▶ aim point ─▶ turret yaw ─▶ (click) weapon ─▶ shells ─▶ target hits
```

## Controls (third-person)
- **W / ↑** gas, **S / ↓** reverse/brake, **A/D or ←/→** steer (differential tracks).
- **Mouse** aim turret, **Left-click** fire (cooldown).
- **R** reset.

## Error handling & edge cases
- Missing named node after load → explicit console error naming the missing part
  (don't silently no-op the animation).
- GLB load failure / served via `file://` → on-screen hint to use `start_server.bat`.
- Speed = 0 → tracks/belt idle (no scroll), wheels still.
- Reverse → wheels and belt scroll backward.
- Pure pivot turn (one side forward, other back) → belts scroll opposite directions.
- Fire on cooldown → ignored (no shell).
- Shell lifetime cap → despawn after max range/time.

## Verification strategy
- `node --check index.html`-embedded JS (extract or lint) and a **headless node**
  logic test (drive math, differential sign, cooldown, shell travel) — like the
  biplane (`scratchpad/flighttest`).
- `npx @gltf-transform/cli inspect assets/tank.glb` to confirm named nodes/materials.
- **Offline render** (numpy point-splat, as used for the biplane) to confirm front
  orientation, turret location, and belt/wheel segmentation before wiring aim.
- The preview sandbox (software GL) cannot render ~1M tris live; final visual check
  is the user in a real browser via `start_server.bat`.

## Scope guardrails (out of scope for v0.1 — future phases)
- Enemy AI / opponent tanks, terrain/heightfield, multiplayer.
- Barrel pitch aiming (default out; can add later).
- Model decimation for load-time (optional later optimization).
- Damage model / health / respawn beyond simple target destruction.

## Risks
- **Turret/barrel segmentation** could be imperfect (thin barrel, merged mass).
  Fallback: ship v0.1 with turret as one rotating unit (no separate barrel pitch),
  or — if the turret can't be cleanly split — a hull-forward cannon with
  camera-relative aim, and add turret yaw once segmentation is solid.
- **Belt arc-length UVs**: the loop is an oval (top/bottom runs + end curves);
  parameterization must stay continuous around the seam. Mitigate by parameterizing
  on angle about the loop center and validating the seam in the inspect/render step.
- **Performance/load**: 40 MB GLB loads slowly. Acceptable for v0.1; decimation is a
  named later step.
